# UFC Fight Predictor — Plan

## Context

The repo currently has a working ufcstats domain (HTML scraper + dashboard for upcoming UFC fights with side-by-side fighter stat comparison). The user wants to extend this into a fight outcome predictor:

1. **Backfill** every historical UFC fight + per-round stats + per-fighter stats into a real database (currently only on-disk JSON cache).
2. **Add a second data source**: moneyline odds from `the-odds-api.com` (free tier, 500 req/month).
3. **Train an XGBoost model** on the historical data to predict win probability for any matchup.
4. **Surface predictions in the dashboard**: pick + confidence + research-only edge vs the market line, plus a track-record page and admin retrain trigger.

Why now: the ufcstats domain proves the data plumbing works end-to-end. The next step turns it into something with predictive output (and a measurable track record over time).

## Phase 1 — On-demand fighter backfill

**Goal:** Users backfill fighter history one matchup at a time, from the dashboard. The DB grows organically with what people actually look at — not 25 years of obscure prelim fighters nobody cares about. Each fighter's backfill is timestamped so we can detect staleness when they fight again.

### Backfill states

A fighter is in one of these states (stored on `fighter_backfill_state.status`):

| State | Meaning | UI affordance |
|---|---|---|
| `none` | Never backfilled | "Load history" button (red badge) |
| `current` | `history_count_at_backfill == fighter.historyCount` and snapshot recent | ✓ Loaded (green badge) |
| `stale_count` | Fighter has new fights since backfill (`historyCount` increased) | "Update — N new fights" button (yellow badge) |
| `stale_stats` | Backfill is older than 7 days but no new fights | Quietly auto-refresh on next view; no UI action |
| `in_progress` | Backfill running now | Spinner |
| `failed` | Last attempt errored | "Retry" button + error tooltip |

The "stale_count" detection is the smart bit: each time a user views a card containing a backfilled fighter, the API hits `/api/ufcstats/fighter/:id` (which is itself cached for 7 days), compares `historyCount` against `history_count_at_backfill`, and flags if behind. This costs nothing extra because the fighter detail call already happens for the comparison sheet.

### Backfill mechanics for one fighter

When the user clicks "Load history" (or "Update"):

1. `POST /api/predict/backfill/fighter/:id` → returns immediately with `{job_id, status: 'in_progress'}`.
2. Background worker (in-process — no separate scheduler):
   - `GET /api/ufcstats/fighter/:id` (cached) → upsert `fighters` row + write current `fighter_stat_snapshots` row with `snapshot_at = now()`.
   - Walk `history[]` array — for each fight that's NOT already in `fights`, fetch `GET /api/ufcstats/fight/:id` → upsert `fights` + `fight_results` (both fighters) + `fight_round_stats`.
   - For each new fight encountered, also queue the **opponent** for stat-snapshot creation (no full backfill of opponent's history — just one snapshot of their stats at that point so we have features when this fighter is the "training row").
   - Update `fighter_backfill_state` with `history_count_at_backfill = fighter.historyCount`, `last_known_fight_id = history[0].fightId`, `status = 'current'`, `last_backfilled_at = now()`.
3. UI polls `GET /api/predict/backfill/job/:job_id` every 1s to show progress.

**Update path** (stale_count): same flow, but only fetches fights newer than `last_known_fight_id`. Typical update = 1-3 new fights = a few seconds.

**Bulk seed (Phase 3 prerequisite):** The training model needs a corpus before it's useful. To bootstrap without forcing the user to click 200 times, add admin buttons on `/admin/predict-train`:

- **"Seed: last 5 years (UFC only)"** — walks `events/completed?page=N` filtering `promotion='ufc'` and `event.date >= now() - 5y`. Queues every distinct fighter encountered for backfill via the same per-fighter flow. Approx 250 events × ~12 fights × ~1.7 unique new fighters per fight ≈ ~5,000 fighter-backfills (with massive de-duplication, likely <800 unique fighters). At 30 req/min this takes 30-60 minutes.
- **"Seed: PPVs only (last 5 years)"** — narrower (~60 events), finishes in ~15 minutes; useful for fast iteration on feature engineering.
- **"Seed: in-window cards"** — backfills only fighters appearing on cards within the next 30 days; smallest, finishes in <5 minutes.

All three reuse the same per-fighter backfill machinery under the hood — they just generate different work queues. Progress is tracked in the same `fighter_backfill_state` table so you can resume after Ctrl-C and see real-time counts on the admin page.

### Files to create / modify

- `packages/db/src/schema.ts` — add `fighter_backfill_state` table (defined in Phase 0).
- `apps/api/src/routes/backfill.ts` (NEW) — `POST /api/predict/backfill/fighter/:id`, `POST /api/predict/backfill/event/:eventId` (both fighters in every matchup), `GET /api/predict/backfill/job/:job_id`, `GET /api/predict/backfill/state/:fighterId`, `POST /api/predict/backfill/seed` (admin — seeds training set).
- `apps/api/src/services/backfill-worker.ts` (NEW) — in-process job queue with `Map<jobId, {status, progress, error}>`. No external scheduler — jobs run on the same process. On crash, jobs disappear; users see "failed" and re-click. Acceptable for v1.
- `apps/web/src/app/(dashboard)/upcoming/event-fight-card.tsx` (MODIFIED) — fetch backfill state for both fighters on each row; show colored badge per fighter (red/yellow/green dot next to name).
- `apps/web/src/app/(dashboard)/upcoming/compare-sheet.tsx` (MODIFIED) — corner cells show backfill state + button. Disable "View prediction" until both fighters are at least `current` (ML can't predict on `none`-state fighters, since we have no point-in-time stats for opponents in their history).
- `apps/web/src/app/(dashboard)/admin/predict-train/page.tsx` (MODIFIED) — adds the two seed buttons.

### Why this is better than bulk backfill

- Tens of GB of irrelevant data (1990s prelim fighters who never fought again) never hits the DB.
- The hot fighters that 99% of card-viewers care about get loaded organically.
- Staleness detection is automatic — no nightly cron needed.
- A user clicking around the next 2-3 cards naturally builds the corpus the model needs.
- The seed-button gives us a known-good baseline without forcing any single user to do bulk work.

## Phase 2 — Odds domain plugin

**Goal:** New `domains/odds-mma/` plugin that fetches moneyline odds and snapshots them into `odds_snapshots`.

Files to create:

- `domains/odds-mma/package.json` — same pattern as `domains/ufcstats/`.
- `domains/odds-mma/src/{config,interceptor,index}.ts` — domain registration boilerplate (no traffic capture, `browserRequired: false` everywhere).
- `domains/odds-mma/src/cache.ts` — patterned on `domains/ufcstats/src/cache.ts` but with two buckets: `oddsList` (1 hour TTL) and `oddsHistorical` (1 year, in case any historical lookup is added later).
- `domains/odds-mma/src/routes.ts` — three routes:
  - `GET /api/odds-mma/upcoming` — proxies `/v4/sports/mma_mixed_martial_arts/odds?regions=us&markets=h2h`. Cached 1h.
  - `GET /api/odds-mma/snapshot` — same fetch, but ALSO writes one row per fighter per bookmaker into `odds_snapshots` keyed to current time. Used by the nightly cron.
  - `GET /api/odds-mma/match` — internal helper that takes an event_id from our DB + the odds API response → fuzzy-matches by event date + fighter names → returns the join.
- Reads `ODDS_API_KEY` from `.env` (add to `.env.example`).
- In `apps/api/src/register-domains.ts`: `registerDomain(oddsMma)` + `registerRateLimit('api.the-odds-api.com', { maxPerMinute: 5, retryOn429: 1 })`.

**Budget plan for 500/mo:** the-odds-api MMA endpoint returns ALL upcoming MMA odds in one response. So:
- Daily cron at 09:00 → `/snapshot` → 1 call → 30/mo.
- Manual `/upcoming` from dashboard → cached 1h, so worst case 24/mo.
- Headroom for testing/dev: ~440 calls.

**Fuzzy matching:** odds API names fighters like `"Khamzat Chimaev"` while ufcstats has `"Khamzat Chimaev"` exactly — but watch for `"Sean O'Malley"` vs `"Sean OMalley"`, accents, suffixes. Strategy: normalize both sides (lowercase, strip non-alnum), then key on `(event_date_iso, sorted_fighter_pair)`. Document mismatches into a `unmatched_odds` table for manual review rather than dropping silently.

## Phase 3 — Python ML service extension

**Goal:** Train XGBoost in Python, expose train + predict via JSON-RPC over the existing IPC bridge.

Files to modify / create:

- `services/python/requirements.txt` — add `xgboost`, `scikit-learn`, `pandas`, `numpy`, `psycopg[binary]`.
- `services/python/ml/__init__.py`
- `services/python/ml/db.py` — `psycopg` connection pool from `DATABASE_URL`.
- `services/python/ml/features.py` — given a fight + both fighters, build the feature row. **All features are computed AS-OF the target fight's date** by aggregating `fight_results` rows where `event.date < target_fight.date` (strict inequality — no leakage). Where data is missing (e.g. fighter has 0 prior UFC fights, or `dob` is null), feature is set to `NaN` and XGBoost handles it natively.

  **Statistical differentials** (signed: A − B):
  - `slpm_diff` — significant strikes landed per minute (career-to-date)
  - `str_acc_diff`, `sapm_diff`, `str_def_diff` — striking accuracy / absorbed / defense
  - `td_avg_diff` — takedowns per 15 min
  - `td_acc_diff`, `td_def_diff` — takedown accuracy / defense
  - `sub_avg_diff` — submission attempts per 15 min

  **Physicals**:
  - `height_diff`, `reach_diff` (inches)
  - `age_diff` (years at fight date — computed from `dob`)

  **Stance interaction** (categorical, one-hot):
  - `stance_match` ∈ {`orthodox_orthodox`, `orthodox_southpaw`, `southpaw_orthodox`, `southpaw_southpaw`, `switch_involved`, `unknown`}.
  - Note: `orthodox_vs_southpaw` is asymmetric in MMA; southpaws statistically over-perform against orthodox, so we keep direction.

  **Experience**:
  - `ufc_fight_count_a`, `ufc_fight_count_b`, `ufc_fight_count_diff` — number of completed UFC fights as of fight date (not just record total — keeps "career-fight-count vs fight-count-in-this-promotion" distinct).
  - `ufc_debut` boolean — true if either fighter has zero UFC fights pre-this-fight (debutants behave very differently from veterans).

  **Recent form** (rolling windows, computed at point in time):
  - `wins_last_3_diff`, `wins_last_5_diff` — count of wins in last N fights
  - `loss_streak_a`, `loss_streak_b` — consecutive losses immediately preceding this fight
  - `coming_off_loss` boolean per fighter
  - `days_since_last_fight_a`, `days_since_last_fight_b`, `days_since_last_fight_diff` — proxy for ring rust + recovery time
  - `long_layoff_a`, `long_layoff_b` — boolean, > 365 days since last fight (proxy for likely injury layoff, since we have no direct injury feed)
  - `finish_rate_diff` — (wins by KO/TKO + wins by SUB) / total wins
  - `decision_rate_diff` — wins by decision / total wins (proxy for "grinder" style)
  - `time_in_cage_a`, `time_in_cage_b` — cumulative seconds across all prior fights (career mileage; a 35-year-old with 8 minutes total is different from a 35-year-old with 4 hours total)

  **Weight class**:
  - `weight_class_change` boolean — true if `target_fight.weight_class` differs from the fighter's most recent fight's weight class
  - `same_weight_class_count` — number of fights in this exact weight class as of date (proxy for division acclimation)

  **Damage taken (injury proxy — best we can do without medical data)**:
  - `damage_index_a`, `damage_index_b` — sum of (sig strikes absorbed in last 3 fights, weighted heavier for KO/TKO losses). High damage index recently → likely diminished. Documented as a proxy with known weaknesses; expect modest predictive value.

  **Notable absences** (call them out explicitly so we don't pretend the model is more comprehensive than it is):
  - No actual injury reports — no medical, no fight-week withdrawal patterns, no public injury news scraping. The `damage_index` and `long_layoff` flags are the only proxies.
  - No camp / coach / training partner features — would need an external source.
  - No betting line movement (line shifts can be informative — could be added later if odds API gives historical line history; the free tier doesn't).
  - No physical conditioning metrics (gas tank, cardio) — partially captured by `time_in_cage` and `decision_rate`.

- `services/python/ml/train.py` — load completed fights, build feature matrix, **chronological** 80/20 split by event date (NOT random — would leak future into past). Train XGBoost classifier with early stopping on the chronological val set. XGBoost handles `NaN` natively, so missing-feature rows aren't dropped. Save model + metadata to `data/models/<UTC-iso>.json` (XGBoost native format), insert row in `model_versions`. Persist feature-importance scores to the same row for inspection on `/admin/predict-train`.
- `services/python/ml/predict.py` — load latest active model, take `(fighter_a_id, fighter_b_id, fight_date)` → build feature row using same `features.py` logic with `as_of_date = fight_date` → predict win prob.
- `services/python/worker.py` — extend handler dispatch with three new methods:
  - `ml.train` → returns `{model_version, accuracy, log_loss, brier_score, roc_auc, training_set_size, top_features: [(name, importance), ...]}`.
  - `ml.predict` → returns `{predicted_winner_id, win_prob, model_version, contributing_features: [(name, value, shap), ...]}` — SHAP values let us show "why" the model picked a fighter.
  - `ml.list_models` → returns last N rows of `model_versions`.
- `services/python/test_ml.py` — pytest covering edge cases:
  - Fighter with 0 prior fights → all "career-to-date" features should be `NaN`, not 0.
  - Missing `dob` → `age_diff = NaN`.
  - Stance match generates correct one-hot.
  - Point-in-time leakage test: `as_of_date = 2020-01-01` MUST not see any fight with `event.date >= 2020-01-01` (computes feature, asserts it equals the same fighter's feature one day before their next fight).
  - Weight class change detection: fighter who fought 3 fights at LW then moves to WW → `weight_class_change = true`.

### Metrics + scoring

Tracked per training run, stored in `model_versions`:

| Metric | Why it matters |
|---|---|
| **Log loss** | Standard classification calibration metric. Random = 0.693. Realistic target for MMA: 0.62-0.66. |
| **Brier score** | Squared error of predicted probability vs outcome. Best metric for "is the predicted 65% actually a 65% over many trials?" — directly informs how trustable our edge calculations are. |
| **Accuracy** | Picks correct winner (threshold 0.5). Vegas closing-line accuracy on MMA is ~67%; if we beat that consistently on a chronological holdout, that's the headline. |
| **ROC AUC** | Ranking quality independent of threshold. |
| **Simulated ROI vs market** | For training set fights where we have odds: simulated ROI of "track whenever model edge > 5%". This is research-only until leakage audits, baselines, and market-gated validation pass. Stored separately because it depends on having odds data. |

**Calibration plot** (separate from training metrics, generated each train run): bin predictions by predicted probability (10 buckets), show actual win rate per bucket. A well-calibrated model produces a near-diagonal line; a confident-but-wrong model bows below. The `/predictions` track-record page renders this same plot from real predictions over time.

Realistic outcome targets, stated up front so we know if the model is actually working:
- Beat naive baseline (always pick higher-experience fighter): YES — easy.
- Match closing line accuracy (~67%): plausible after seeding 5+ years of data.
- Validated market edge: hard. Markets are efficient; positive simulated ROI is only useful after leak-free baselines, sufficient odds coverage, and market-gated validation. Negative simulated ROI does not mean the model is bad; it means the market priced the same factors first.

## Out of scope (intentional)

- Prop bets / over-under / method-of-victory markets — moneyline only for v1.
- Multi-bookmaker arbitrage detection — single best-line per fighter is enough for the edge calc.
- Streaming odds / live in-fight predictions — strictly pre-fight static snapshots.
- A/B testing model versions in production — only one "active" model at a time; older models stay queryable in `model_versions` but aren't served.
- Auto-betting — predictions are advisory only.

## Risks & open questions

1. **Fighter name fuzzy-match accuracy on odds API.** Plan documents `unmatched_odds` table for manual review. If miss rate is high (>5%), need alias table. Decide after first week of snapshots.
2. **Point-in-time leakage.** Easy to accidentally use post-fight stats in features. The Phase 3 `features.py` MUST aggregate from `fight_results` rows with `event.date < target_fight.date`. Has a dedicated unit test (`test_ml.py::test_no_leakage`).
3. **the-odds-api free tier rate-limit ceiling.** 5 req/min is tight if multiple devs hammer the dashboard. Cache-first design mitigates, but worth a `?refresh=1` warning toast in the UI.
4. **On-demand backfill leaves opponent stats sparse.** When Fighter A is backfilled, we get full point-in-time stats for A but only a single current snapshot for each of A's past opponents. For ML features that read "opponent stats at time of fight", we have only the current snapshot. Mitigation: when backfilling A, ALSO write a `fighter_stat_snapshots` row for each opponent with `snapshot_at = the fight's event date` — this approximates point-in-time by treating the snapshot as valid up to the next snapshot. Documented in the Phase 1 backfill mechanics step 2 sub-bullet ("queue the opponent for stat-snapshot creation"). Imperfect, but good enough for v1.
5. **Initial training set size depends on user behavior.** Until the user clicks Seed, the model has no data. The seed button is mandatory before clicking Train.
6. **Damage / injury proxy is weak.** The `damage_index` feature combines absorbed strikes + recent KO losses + long layoffs. Real injury data would require an external source (UFC.com news scraping or the like) — out of scope for v1.
7. **No labels for upcoming fights.** Predictions for upcoming events go into `predictions` with `actual_outcome=NULL`; the `/predictions` page joins against `fight_results` once results are backfilled (which now happens automatically when a user views the post-event card and triggers a fighter backfill on either fighter — they share the same fights table).
8. **Ringside events / non-UFC promotion data.** ufcstats.com tracks fights from other promotions in fighter histories (Bellator, ONE, Strikeforce, regional shows). These have stats too. Decision: include them in `fight_results` since they appear in the same scraped table. The `weight_class_change` and `ufc_fight_count` features will need to distinguish UFC-only events — derived by joining `events.name` LIKE 'UFC%' or storing a `promotion` column. Add a `promotion` enum column to the `events` table at schema time to avoid migration pain later.
